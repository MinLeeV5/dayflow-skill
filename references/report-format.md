# Dayflow 结构化总结模板

最终输出一律使用中文，并严格保持下面的章节顺序。

## 固定结构

```markdown
# Dayflow 工作总结（<时间范围>）

数据来源：Dayflow `timeline_cards`（必要时补充 `journal_entries`）

## 目标（含组织与个人）
- 组织目标：
- 个人目标：

## 关键成果（交付物/数据）
- ...

## 关键行动举措（对齐组织目标拆解，需体现延期情况）
- 行动/举措：
  - 对齐目标：
  - 进展/结果：
  - 延期情况：

## 完成情况（成效、问题、风险、措施）
- 成效：
- 问题：
- 风险：
- 措施：

## 工作质量
- ...

## 工时/D（具体人天）
- 总工时：<X> 小时
- 折合人天：<Y> D（按 8 小时/天）

## 自评难易（难/中/易）
- ...

## 备注
- ...
```

## 字段填写规则

### 目标（含组织与个人）

- 优先使用 `journal_entries.intentions` 和 `journal_entries.goals`
- 如果没有，再从时间线的重复主题中推断，并明确标记“以下为推断”
- 要区分组织目标与个人执行/成长目标

### 关键成果（交付物/数据）

- 优先写具体产出，例如：
  - 文档、方案、流程图、代码提交、面试结论、评审结果、发布动作、对外输出
- 只有在提取数据能直接支撑时，才写数量、时长或统计
- 日常沟通本身不算成果，除非它明显推动了工作进展

### 关键行动举措（对齐组织目标拆解，需体现延期情况）

- 按工作流/工作主题分组，不要一张卡片对应一条动作
- 每个动作都要说明：
  - 做了什么
  - 对应支撑了哪个目标
  - 在 Dayflow 中能看到什么进展或结果
  - 是否存在延期、阻塞、返工、待跟进事项等证据
- 如果数据里没有明显延期证据，就写：`未从 Dayflow 数据中看到明确延期证据。`

### 完成情况（成效、问题、风险、措施）

- `成效`：交付价值、推进幅度、闭环情况、协同结果
- `问题`：模糊点、打断、反复修改、沟通成本
- `风险`：未解决事项、依赖风险、等待状态、证据不足项
- `措施`：从观察到的工作轨迹中推导出的下一步动作或缓解措施

### 工作质量

根据以下证据判断工作质量：
- 工作块是否连续
- 跟进深度是否足够
- 产出是否从草稿推进到完成
- 是否存在较多返工
- `metadata` 中是否出现分心或频繁切换上下文的线索

保持基于证据的判断，不要泛泛而谈。

### 工时/D（具体人天）

- 必须使用读取脚本统计出的总量
- `D = total_hours / 8`
- 小时与 D 都要展示，保留 2 位小数

### 自评难易（难/中/易）

使用保守判断：
- `难`：目标模糊度高、并行任务多、上下文切换频繁、协调负担明显
- `中`：执行有深度，但范围相对可控
- `易`：常规推进，模糊度和协作成本都较低

用一句话解释为什么这么评定。

### 备注

这里用于补充限制条件或说明：
- 某些日期没有 Dayflow 覆盖
- 某些字段来自时间线推断而非 journal
- 业务指标无法从 Dayflow 中直接看到
- 与隐私、证据充分性有关的补充说明

## 语气要求

- 简洁、事实优先、中文优先
- 明确区分“直接观察事实”与“基于轨迹的推断”
- 优先使用 `根据 Dayflow 记录`、`从活动轨迹推测` 这类表述
- 只有模式迹象、没有直接证据时，不要写成确定结论
